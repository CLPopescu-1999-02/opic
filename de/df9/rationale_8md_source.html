1 RATIONALE AND ROADMAP {#rationale}
2 ---------------------
3 
4 In most of the programming models, program operates on in memory
5 objects, and operations on off-heap (disk, network) objects are usually
6 afterthoughts. The off-heap object representations are mostly
7 implemented per application basis, such as database, primitive only
8 serialization format (protobuf, thrift, avro), search engine index, or
9 document format like Microsoft Word. Implementing new serialization
10 formats for every new applications is painful, so we decided to create a
11 general framework to rule it all.
12 
13 A general purpose serialization framework can do more than that. We
14 target to serialize all possible data structures. BST, Hash table, B+
15 Tree, suffix array, anything. As you might notice, these are the core
16 data structures to build databases and search engines. Not only should
17 you be able to write a hobby transactional database out of this
18 framework, but you can also build your own NoSQL, multi-layer DB like
19 LevelDB/RocksDB, bitmap based DB like Druid, or experiment new database
20 concepts that no one has seen before. This is one core aspect for
21 ADT (abstract data type) serialization.
22 
23 This concept can be further extend to serialize the general program
24 state (not yet implemented). The motivation comes from bad experiances
25 on using memory aggressive browsers. The browser performance is not
26 satisfactory when having dozens of browser tabs opened, even though the
27 OS should swap out the memory to disk but it just doesn't perform well.
28 Instead, why not serialize the whole Javascript and DOM state out, and
29 restore it back when user requires it? Our initial attempt is build a
30 lambda calculus engine base on the serialization framework, then
31 optimize its performance and write Javascipt compiler to the lambda
32 form.
33 
34 If you also feel excited on this project, your contributions and patches
35 are always welcome.

43 ### Language Choice
44 
45 Before we jump to the serialization abstraction, let me briefly discuss
46 the language choice. This framework is implemented in pure C, following
47 C11 standard and GCC extensions. C11 is chosen for atomic variables; A
48 subset of GCC extensions are carefully picked in which it has the
49 support from clang as well. Other compiler (Visual Studio, Intel C)
50 support are not considered on current stage of development. The use of
51 GCC extensions are necessary because many important language features do
52 not exist in C99/C11 standards. The extensions we use and plan to use
53 are:
54 
55 * constructor function
56 * address alignment
57 * SIMD vector types
58 * may add more if needed.
59 
60 Though breaking portability across multiple compilers could be a
61 problem, we plan to overcome it by using Autoconf configuration
62 to detect compiler supported features before actual compilation.
63 If the feature test failed, it would not try to compile.
64 
65 Many people asked me why didn't I just use C++ instead. There are two
66 main reasons. First, serializing general types is crucial in C++. You
67 have to serialize the internal C++ structures such as vtable (which is
68 implementation defined), having your own memory allocator to restore
69 load the de-serialized C++ type, and probably trace all the method
70 address of an object. That's too much hack. And since the compiler may
71 use different vtable layout on different versions, you have to test
72 against all possible ones! Nope, not a good idea. Second reason relates
73 to how I want this framework to be used. ALL programming languages has C
74 bindings, and that is a perfect place to hide the complicated internals
75 and expose high level API to end users. For instance, expose a
76 programmable database interface to Python that looks like a regular
77 container. As all the states are written in C, it is also easier to pass
78 it to other language's C bindings.
79 
80 Same considerations apply to all other possible candidates, including
81 go, rust, swift, OCaml, Haskell, Java, Scala, etc. Unless you can
82 serialize its internal structure easily with deterministic performance,
83 it won't be taken onto the table.

85 ### Generics (completed)
86 
87 So, we decided to use C. But it is so featureless! How do we balance out
88 the low level control and high level abstraction to make programming
89 less painful? As you might guess, we use macros to create a DSL. The
90 main feature I tired to implement is generic type and interface (also
91 called traits, or typeclass in haskell). See the following example of
92 how it makes language semantics cleaner.
93 
94 ```c
95 serialize((void*)obj, fd);
96 ```
97 
98 A prototype function like above can behave differently with different
99 object instance. If the object has internal object that need to get
100 serialized, simply recursively call `serialize` again with the internal
101 object.
102 
103 For now we only implemented generics (method polymorphism), without
104 inheritance support. Though it is not hard but we won't do it until we
105 really need it. Yet, the generics abstraction is already powerful enough
106 for us to write Haskell Monads and Functors in C. See
107 [monad_example](https://github.com/dryman/opic/tree/master/monad_example)
108 for how it works.

110 ### Serializable Pointer Machine (completed)
111 
112 You can think [pointer machine][pmachine] is a academic synonym to
113 object oriented programming model. It is also a general form of LISP
114 machine which is used by many functional languages.
115 
116 I follow the Schönhage's SMM (Schönhage's Storage Modification Machine)
117 definition of pointer machine, where the machine is formed by number of
118 "nodes" with a inbound link to the root node. Each node have constant
119 number of field; a field could be an outbound pointer to other node, or
120 a primitive type. Only certain operations are permitted: create (new),
121 destroy (free) nodes, follow the pointers, or mutate the fields in a
122 node. This pretty much models general object oriented program behavior,
123 and it can emulate LISP machine as well.
124 
125 This model is also carefully picked for other practical reasons.
126 
127 * Since pointer machine can only use constant size objects, we can
128  group same type of objects in large chunk of array, and can serialize
129  it in batches.
130 * Compare to the model above, RAM model is harder to manage and make
131  it deterministic on serialization operation.
132 
133 The detail explaination is in
134 [pm_serde_example](https://github.com/dryman/opic/tree/master/pm_serde_example).
135 
136 [pmachine]: https://en.wikipedia.org/wiki/Pointer_machine

139 
140 Implements common collection types for good code reuse. In contrast to
141 most real time applications, one major use of this library is for
142 serialization which only require good amortized complexity. Therefore we
143 permit and encourage amortized data structures like Fibonacci heap to be
144 included in this library. This unleashes many powerful data structures
145 that only exist in papers but yet used in real world.
